## Data Model

A few conventions that apply to every table before we start:

- All primary keys are `UUID` generated by the application, not the database. This avoids sequential ID enumeration attacks and works correctly across distributed systems if you ever shard.
- All monetary values are `NUMERIC(12,2)` — never `FLOAT` or `DOUBLE`. Floating point arithmetic on money is a genuine correctness bug, not a style issue. `NUMERIC` is exact decimal arithmetic.
- All timestamps are `TIMESTAMPTZ` (timezone-aware). Your server stores everything in UTC. Display conversion is the client's responsibility.
- Every table has `created_at TIMESTAMPTZ DEFAULT NOW()`. Most have `updated_at` maintained by a trigger.
- Soft deletes use `deleted_at TIMESTAMPTZ NULL`. A null value means the record is active.

## Schema — Auth Module

### `users`
The root identity record. Kept intentionally minimal — only what auth needs.

```sql
CREATE TABLE users (
    id                  UUID            PRIMARY KEY,
    username            VARCHAR(50)     NOT NULL UNIQUE,
    email               VARCHAR(255)    NOT NULL UNIQUE,
    password_hash       VARCHAR(255)    NOT NULL,
    email_verified      BOOLEAN         NOT NULL DEFAULT FALSE,
    is_discoverable     BOOLEAN         NOT NULL DEFAULT FALSE,
    is_active           BOOLEAN         NOT NULL DEFAULT TRUE,
    failed_login_count  INT             NOT NULL DEFAULT 0,
    locked_until        TIMESTAMPTZ     NULL,
    created_at          TIMESTAMPTZ     NOT NULL DEFAULT NOW(),
    updated_at          TIMESTAMPTZ     NOT NULL DEFAULT NOW()
);
```

**Design notes:**
- `username` and `email` uniqueness is enforced at the DB layer, not just application layer. The `409 Conflict` response on registration is driven by catching a `UniqueConstraintViolationException`.
- `failed_login_count` and `locked_until` implement your brute force lockout (5 attempts / 10 minutes) at the DB layer. The application resets `failed_login_count` to 0 on successful login.
- `is_discoverable` lives here rather than `user_profiles` because the Auth module owns it and checks it during access grant creation.

### `revoked_tokens`
Tracks invalidated JWT tokens for synchronous logout.

```sql
CREATE TABLE revoked_tokens (
    id              UUID        PRIMARY KEY,
    user_id         UUID        NOT NULL REFERENCES users(id),
    token_jti       VARCHAR(255) NOT NULL UNIQUE,  -- JWT ID claim
    revoked_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    expires_at      TIMESTAMPTZ NOT NULL            -- token's original expiry
);

CREATE INDEX idx_revoked_tokens_jti ON revoked_tokens(token_jti);
CREATE INDEX idx_revoked_tokens_expires ON revoked_tokens(expires_at);
```

**Design notes:**
- `token_jti` is the JWT `jti` (JWT ID) claim — a unique identifier baked into every token at issuance. On every authenticated request, your filter looks up the `jti` in this table. If found, reject the request.
- The `expires_at` index enables a cleanup job: rows where `expires_at < NOW()` are dead weight (the token would have been rejected by expiry anyway). A nightly job deletes them. Without this, the table grows forever.

---

### `sudo_tokens`
Short-lived step-up authentication tokens for cross-user data access.

```sql
CREATE TABLE sudo_tokens (
    id          UUID        PRIMARY KEY,
    user_id     UUID        NOT NULL REFERENCES users(id),
    token_hash  VARCHAR(255) NOT NULL UNIQUE,  -- hashed, never stored plain
    expires_at  TIMESTAMPTZ NOT NULL,
    used_at     TIMESTAMPTZ NULL,               -- single-use enforcement
    created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_sudo_tokens_user ON sudo_tokens(user_id, expires_at);
```

**Design note:** The sudo token value itself is hashed before storage (same principle as passwords). A compromised DB does not yield valid sudo tokens. The token is also **single-use** — `used_at` is set the first time it's consumed, and subsequent requests with the same token are rejected even if still within the expiry window.

Extended profile data, separate from the auth-critical `users` table.

```sql
CREATE TABLE user_profiles (
```

### `access_grants`
Cross-user data sharing grants.

```sql
CREATE TABLE access_grants (
    id                  UUID        PRIMARY KEY,
    grantor_user_id     UUID        NOT NULL REFERENCES users(id),
    grantee_user_id     UUID        NOT NULL REFERENCES users(id),
    access_level        VARCHAR(20) NOT NULL,    -- READ_ONLY | FULL
    expires_at          TIMESTAMPTZ NOT NULL,
    revoked_at          TIMESTAMPTZ NULL,
    created_at          TIMESTAMPTZ NOT NULL DEFAULT NOW(),

    CONSTRAINT chk_access_level 
        CHECK (access_level IN ('READ_ONLY', 'FULL')),
    CONSTRAINT chk_no_self_grant 
        CHECK (grantor_user_id != grantee_user_id),
    CONSTRAINT chk_expiry_range
        CHECK (expires_at <= created_at + INTERVAL '10 days'
           AND expires_at >= created_at + INTERVAL '1 day')
);

CREATE INDEX idx_access_grants_grantor ON access_grants(grantor_user_id);
CREATE INDEX idx_access_grants_grantee ON access_grants(grantee_user_id, expires_at)
    WHERE revoked_at IS NULL;
```

**Design notes:**
- `chk_no_self_grant` prevents a user granting themselves access — meaningless and a potential abuse vector.
- `chk_expiry_range` enforces the 1–10 day window at the DB layer, not just the API layer.
- The partial index on `grantee_user_id` (with `WHERE revoked_at IS NULL`) means active grant lookups are fast and the index stays small — revoked grants are excluded.

## Schema — Expense Module

### `categories`
User-defined and system-defined expense categories.

```sql
CREATE TABLE categories (
    id              UUID        PRIMARY KEY,
    user_id         UUID        NULL REFERENCES users(id),  -- NULL = system category
    name            VARCHAR(100) NOT NULL,
    description     VARCHAR(255) NULL,
    is_system       BOOLEAN     NOT NULL DEFAULT FALSE,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),

    CONSTRAINT uq_category_name_per_user 
        UNIQUE NULLS NOT DISTINCT (user_id, name)
);

CREATE INDEX idx_categories_user ON categories(user_id) WHERE user_id IS NOT NULL;
```

**Design notes:**
- `user_id = NULL` means it's a system-defined category visible to all users (GROCERIES, DINING, TRANSPORT etc.). User-created categories are private.
- `UNIQUE NULLS NOT DISTINCT (user_id, name)` is a PostgreSQL 15+ feature. It means two system categories (both with `user_id = NULL`) cannot share the same name. Without this, standard `UNIQUE` would allow duplicate system category names because `NULL != NULL` in standard SQL uniqueness checks.

### `expenses`
The core table. Partitioned by year.

```sql
CREATE TABLE expenses (
    id                      UUID            NOT NULL,
    user_id                 UUID            NOT NULL REFERENCES users(id),
    amount                  NUMERIC(12,2)   NOT NULL,
    merchant_name           VARCHAR(255)    NOT NULL,
    expense_date            DATE            NOT NULL,
    notes                   TEXT            NULL,
    payment_method          VARCHAR(50)     NULL,
    bank_account_id         UUID            NOT NULL,  -- mandatory, cash account if cash
    source                  VARCHAR(20)     NOT NULL,  -- MANUAL | BANK_IMPORT
    bank_status             VARCHAR(20)     NULL,      -- PENDING | POSTED | NULL for manual
    external_transaction_id VARCHAR(255)    NULL,      -- Basiq transaction ID
    idempotency_key         UUID            NULL,      -- client-supplied, manual entries only
    ai_categorised          BOOLEAN         NOT NULL DEFAULT FALSE,
    is_merged               BOOLEAN         NOT NULL DEFAULT FALSE,
    merged_from             UUID[]          NULL,      -- immutable after set
    deleted_at              TIMESTAMPTZ     NULL,
    created_at              TIMESTAMPTZ     NOT NULL DEFAULT NOW(),
    updated_at              TIMESTAMPTZ     NOT NULL DEFAULT NOW(),

    PRIMARY KEY (id, expense_date),   -- partition key must be in PK

    CONSTRAINT chk_source 
        CHECK (source IN ('MANUAL', 'BANK_IMPORT')),
    CONSTRAINT chk_bank_status 
        CHECK (bank_status IN ('PENDING', 'POSTED') OR bank_status IS NULL),
    CONSTRAINT chk_amount_positive 
        CHECK (amount > 0),
    CONSTRAINT uq_external_transaction 
        UNIQUE (user_id, external_transaction_id)   -- prevents Basiq duplicates at DB layer
) PARTITION BY RANGE (expense_date);
```

**Important design note on the composite PK:**
PostgreSQL requires the partition key (`expense_date`) to be part of the primary key on a partitioned table. This is a real constraint, not a design choice. The application always has the `expense_date` when looking up by ID (it's returned with every expense), so this is workable. Alternatively you could use a unique index on `id` alone across partitions — but the composite PK is the more standard approach.

**Immutability trigger for `merged_from`:**
```sql
CREATE OR REPLACE FUNCTION enforce_merged_from_immutable()
RETURNS TRIGGER AS $$
BEGIN
    IF OLD.merged_from IS NOT NULL AND NEW.merged_from IS DISTINCT FROM OLD.merged_from THEN
        RAISE EXCEPTION 'merged_from is immutable once set';
    END IF;
    IF OLD.is_merged = TRUE AND NEW.is_merged = FALSE THEN
        RAISE EXCEPTION 'is_merged cannot be unset once true';
    END IF;
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_merged_from_immutable
    BEFORE UPDATE ON expenses
    FOR EACH ROW EXECUTE FUNCTION enforce_merged_from_immutable();
```

### `expense_categories`
Junction table — one expense to many categories.

```sql
CREATE TABLE expense_categories (
    expense_id      UUID        NOT NULL,
    expense_date    DATE        NOT NULL,  -- denormalised for partition alignment
    category_id     UUID        NOT NULL REFERENCES categories(id),
    weight_amount   NUMERIC(12,2) NOT NULL,  -- pre-computed even split
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),

    PRIMARY KEY (expense_id, expense_date, category_id),
    FOREIGN KEY (expense_id, expense_date) 
        REFERENCES expenses(id, expense_date)
);

CREATE INDEX idx_expense_categories_category 
    ON expense_categories(category_id, expense_date);
```

**Design note on `expense_date` denormalisation:**
Because `expenses` is partitioned by `expense_date`, the foreign key must include `expense_date`. This means `expense_categories` stores it too — technically denormalised, but a necessary consequence of partition-aware foreign keys in PostgreSQL. The application always has this value when writing categories (it just created the expense), so there's no lookup overhead.

### `expense_idempotency_keys`
Prevents duplicate expense creation from client retries.

```sql
CREATE TABLE expense_idempotency_keys (
    idempotency_key     UUID        PRIMARY KEY,
    user_id             UUID        NOT NULL REFERENCES users(id),
    expense_id          UUID        NOT NULL,
    expense_date        DATE        NOT NULL,
    created_at          TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    expires_at          TIMESTAMPTZ NOT NULL DEFAULT NOW() + INTERVAL '24 hours'
);

CREATE INDEX idx_idempotency_user 
    ON expense_idempotency_keys(user_id, expires_at);
```

**Design note:** Idempotency keys expire after 24 hours. A client retrying a request more than 24 hours later will create a new expense rather than get the original back — this is acceptable and standard behaviour (Stripe uses the same approach). A cleanup job removes expired keys nightly.

## Schema — Target Module

### `expense_targets`

```sql
CREATE TABLE expense_targets (
    id              UUID            PRIMARY KEY,
    user_id         UUID            NOT NULL REFERENCES users(id),
    target_type     VARCHAR(20)     NOT NULL,       -- CATEGORY | TOTAL
    category_id     UUID            NULL REFERENCES categories(id),
    amount          NUMERIC(12,2)   NOT NULL,
    period_type     VARCHAR(20)     NOT NULL DEFAULT 'MONTHLY',
    period_year     INT             NOT NULL,
    period_month    INT             NOT NULL,
    deleted_at      TIMESTAMPTZ     NULL,
    created_at      TIMESTAMPTZ     NOT NULL DEFAULT NOW(),
    updated_at      TIMESTAMPTZ     NOT NULL DEFAULT NOW(),

    CONSTRAINT chk_target_type 
        CHECK (target_type IN ('CATEGORY', 'TOTAL')),
    CONSTRAINT chk_category_required_for_type
        CHECK (
            (target_type = 'CATEGORY' AND category_id IS NOT NULL) OR
            (target_type = 'TOTAL'    AND category_id IS NULL)
        ),
    CONSTRAINT chk_amount_positive 
        CHECK (amount > 0),
    CONSTRAINT chk_period_month 
        CHECK (period_month BETWEEN 1 AND 12),
    CONSTRAINT uq_one_target_per_category_per_period
        UNIQUE NULLS NOT DISTINCT (user_id, target_type, category_id, 
                                    period_year, period_month)
        WHERE deleted_at IS NULL
);

CREATE INDEX idx_targets_user_period 
    ON expense_targets(user_id, period_year, period_month) 
    WHERE deleted_at IS NULL;
```

**Design note on the unique constraint:** `uq_one_target_per_category_per_period` prevents a user creating two DINING targets for April 2026. The `WHERE deleted_at IS NULL` partial constraint means soft-deleted targets don't block creating a new one for the same period.

### `bank_accounts`

```sql
CREATE TABLE bank_accounts (

    user_id                 UUID        NOT NULL REFERENCES users(id),
    institution_name        VARCHAR(100) NOT NULL,
    account_name            VARCHAR(100) NOT NULL,
    account_number_masked   VARCHAR(20)  NULL,

    is_system_account       BOOLEAN     NOT NULL DEFAULT FALSE,  -- TRUE for cash

    last_synced_at          TIMESTAMPTZ NULL,
```

**Design notes:**
- `bitwarden_secret_id` stores only the reference key used to retrieve the OAuth refresh token from Bitwarden at runtime. The token value never enters your database.
- `uq_one_cash_per_user` is a partial unique constraint — it only enforces uniqueness when `is_system_account = TRUE`. A user can have many regular bank accounts but only one cash account.

### `raw_bank_transactions`
Append-only, hash-chained immutable store.

```sql
CREATE TABLE raw_bank_transactions (
    id                      UUID        PRIMARY KEY,
    user_id                 UUID        NOT NULL REFERENCES users(id),
    bank_account_id         UUID        NOT NULL REFERENCES bank_accounts(id),
    external_transaction_id VARCHAR(255) NOT NULL,
    raw_payload             JSONB       NOT NULL,   -- verbatim Basiq response
    ingested_at             TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    content_hash            VARCHAR(64) NOT NULL,   -- SHA-256 of this row's content
    chain_hash              VARCHAR(64) NOT NULL,   -- SHA-256(content_hash + prev chain_hash)
    previous_record_id      UUID        NULL REFERENCES raw_bank_transactions(id),
    processing_status       VARCHAR(20) NOT NULL DEFAULT 'PENDING',

    CONSTRAINT chk_processing_status
        CHECK (processing_status IN ('PENDING', 'PROCESSED', 'FAILED')),
    CONSTRAINT uq_external_per_user
        UNIQUE (user_id, external_transaction_id)
);

CREATE INDEX idx_raw_transactions_user_status 
    ON raw_bank_transactions(user_id, processing_status) 
    WHERE processing_status = 'PENDING';
```

**Append-only enforcement trigger:**
```sql
CREATE OR REPLACE FUNCTION prevent_raw_transaction_modification()
RETURNS TRIGGER AS $$

    RAISE EXCEPTION 
        'raw_bank_transactions is append-only. Updates and deletes are forbidden.';
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_raw_transactions_append_only
    BEFORE UPDATE OR DELETE ON raw_bank_transactions
    FOR EACH ROW EXECUTE FUNCTION prevent_raw_transaction_modification();
```

## Schema — Job Queue and Dead Letter

### `job_queue`

```sql
CREATE TABLE job_queue (
    id              UUID        PRIMARY KEY,
    job_type        VARCHAR(50) NOT NULL,
    user_id         UUID        NULL REFERENCES users(id),
    payload         JSONB       NOT NULL,
    status          VARCHAR(20) NOT NULL DEFAULT 'PENDING',
    retry_count     INT         NOT NULL DEFAULT 0,
    max_retries     INT         NOT NULL DEFAULT 10,
    next_retry_at   TIMESTAMPTZ NULL,
    picked_up_at    TIMESTAMPTZ NULL,
    completed_at    TIMESTAMPTZ NULL,
    last_error      TEXT        NULL,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),

    CONSTRAINT chk_job_type
        CHECK (job_type IN (
            'BANK_SYNC', 
            'NORMALISE_TRANSACTION', 
            'AI_CATEGORISE',
            'ARCHIVE_PARTITION',
            'CREATE_PARTITION'
        )),
    CONSTRAINT chk_status
        CHECK (status IN (
            'PENDING', 'PROCESSING', 'COMPLETED', 'FAILED', 'DEAD_LETTER'
        ))
);

-- The critical index for the FOR UPDATE SKIP LOCKED pattern
CREATE INDEX idx_job_queue_polling 
    ON job_queue(job_type, status, next_retry_at) 
    WHERE status = 'PENDING';

-- Prevent duplicate active sync jobs per user
CREATE UNIQUE INDEX idx_one_active_sync_per_user
    ON job_queue(user_id, job_type)
    WHERE job_type = 'BANK_SYNC' 
      AND status IN ('PENDING', 'PROCESSING');
```

**Design note on `idx_one_active_sync_per_user`:** This partial unique index enforces the "no concurrent sync for same user" rule at the DB layer. The API's `409 Conflict` check is now backed by a DB constraint — even if two requests arrive simultaneously and both pass the application-level check, only one insert will succeed.

## Materialized Views

### `mv_monthly_expense_summary`
Drives dashboard queries and the Target module's prediction context.

```sql
CREATE MATERIALIZED VIEW mv_monthly_expense_summary AS
SELECT
    e.user_id,
    DATE_TRUNC('month', e.expense_date)     AS period_month,
    ec.category_id,
    c.name                                  AS category_name,
    COUNT(DISTINCT e.id)                    AS transaction_count,
    SUM(ec.weight_amount)                   AS total_amount,
    AVG(ec.weight_amount)                   AS avg_amount,
    MIN(e.expense_date)                     AS first_transaction_date,
    MAX(e.expense_date)                     AS last_transaction_date
FROM expenses e
JOIN expense_categories ec ON ec.expense_id = e.id 
                           AND ec.expense_date = e.expense_date
JOIN categories c ON c.id = ec.category_id
JOIN partition_registry pr ON pr.year = EXTRACT(YEAR FROM e.expense_date)
WHERE e.deleted_at IS NULL
  AND pr.status = 'ACTIVE'          -- excludes archived partitions automatically
GROUP BY e.user_id, DATE_TRUNC('month', e.expense_date), 
         ec.category_id, c.name;

CREATE UNIQUE INDEX idx_mv_monthly_summary 
    ON mv_monthly_expense_summary(user_id, period_month, category_id);
```

### `mv_merchant_summary`
Drives "group by merchant" queries.

```sql
CREATE MATERIALIZED VIEW mv_merchant_summary AS
SELECT
    e.user_id,
    e.merchant_name,
    DATE_TRUNC('month', e.expense_date)     AS period_month,
    COUNT(*)                                AS transaction_count,
    SUM(e.amount)                           AS total_amount
FROM expenses e
JOIN partition_registry pr ON pr.year = EXTRACT(YEAR FROM e.expense_date)
WHERE e.deleted_at IS NULL
  AND pr.status = 'ACTIVE'
GROUP BY e.user_id, e.merchant_name, DATE_TRUNC('month', e.expense_date);

CREATE UNIQUE INDEX idx_mv_merchant_summary 
    ON mv_merchant_summary(user_id, period_month, merchant_name);
```

### Refresh strategy
```sql
-- Called after every expense write at current scale
-- Switch to scheduled (every 60s) if write rate increases
REFRESH MATERIALIZED VIEW CONCURRENTLY mv_monthly_expense_summary;
REFRESH MATERIALIZED VIEW CONCURRENTLY mv_merchant_summary;
```

`CONCURRENTLY` means the refresh does not lock the view for reads — queries continue being served from the old data while the refresh runs. This is what makes the bounded staleness approach work without blocking your read traffic.

## Index Strategy Summary

The indexes that will matter most at query time:

```sql
-- Expense queries (most frequent endpoint)
CREATE INDEX idx_expenses_user_date 
    ON expenses(user_id, expense_date DESC) 
    WHERE deleted_at IS NULL;

CREATE INDEX idx_expenses_user_merchant 
    ON expenses(user_id, merchant_name) 
    WHERE deleted_at IS NULL;

CREATE INDEX idx_expenses_user_account 
    ON expenses(user_id, bank_account_id) 
    WHERE deleted_at IS NULL;

-- External transaction lookup (duplicate detection)
CREATE INDEX idx_expenses_external_id 
    ON expenses(user_id, external_transaction_id) 
    WHERE external_transaction_id IS NOT NULL;

-- Category lookup
CREATE INDEX idx_categories_system 
    ON categories(name) 
    WHERE is_system = TRUE;
```

All of these are **partial indexes** (with a `WHERE` clause). A partial index only includes rows matching the condition, making it smaller and faster than a full-table index. An index on only non-deleted expenses is a fraction of the size of an index on all expenses — and it's the only index that ever gets used for live queries.

#### Why `expense_id` Alone Is Not Enough

This is not a design choice — it is a **PostgreSQL constraint on partitioned tables**.

When you partition a table, PostgreSQL requires that the partition key column (`expense_date`) is part of the primary key. This is because PostgreSQL needs the partition key to know *which physical partition* a row lives in. Without it, a foreign key lookup would require scanning every partition to find the row, which defeats the purpose of partitioning entirely.

So your `expenses` primary key becomes `(id, expense_date)` rather than just `(id)`.

The consequence cascades: **any foreign key that references `expenses` must include all columns of the referenced key**. You cannot have a foreign key to just `(id)` if the primary key is `(id, expense_date)`. PostgreSQL will reject the table definition at creation time with an error like:

```
ERROR: there is no unique constraint matching given keys for 
referenced table "expenses"
```

So `expense_categories` is forced to carry `expense_date` in its foreign key:

```sql
FOREIGN KEY (expense_id, expense_date) 
    REFERENCES expenses(id, expense_date)
```

This is not optional or stylistic. It is the price of partitioning.

**The practical implication:** your application always has `expense_date` available when inserting into `expense_categories` because it just created or retrieved the parent expense. There is no extra lookup — you already have the value. The denormalisation is a storage redundancy, not a query-time cost.

**If you find this constraint too painful in practice**, the alternative is to drop the native PostgreSQL partitioning and instead manage partitioning manually — separate tables per year with a view across them. You lose the transparent query routing but regain simple single-column primary keys. At your data volume (roughly 180,000 rows over 5 years), native partitioning's performance benefit is marginal anyway. It is worth knowing this trade-off exists.

#### Enforcing Refresh at the Interface Level

The instinct is right but the location is wrong, and the reason matters.

### Why the interface is the wrong place

The service interface belongs to the **domain layer** — it expresses business intent. `createExpense`, `importBankTransaction`, `softDeleteExpense` describe what the system *does* from a business perspective.

Materialized view refresh is an **infrastructure concern** — it is a PostgreSQL-specific mechanism for maintaining a read cache. If you put refresh logic in the interface, you are leaking infrastructure knowledge into the domain contract. The interface now implicitly requires any implementation to know about and manage a PostgreSQL materialized view. A mock implementation used in tests, or a future `HttpExpenseService` that calls a remote microservice, would have to implement refresh behaviour that is meaningless to them.

The rule from earlier: domain interfaces must not import or imply infrastructure dependencies.

### Where it should live instead

There are three good options, each with a different trade-off:

**Option A — In the implementation directly**

The `PostgresExpenseService` implementation calls refresh after every write. The interface knows nothing about it.

```java
@Service
public class PostgresExpenseService implements ExpenseService {

    @Override
    @Transactional
    public Expense createExpense(UUID userId, CreateExpenseCommand command) {
        Expense expense = buildExpense(command);
        expenseRepository.save(expense);
        categoryRepository.saveAll(buildCategories(expense, command));
        
        // Infrastructure concern lives here, in the implementation
        materializedViewRefreshService.refresh(
            "mv_monthly_expense_summary", 
            "mv_merchant_summary"
        );
        return expense;
    }
}
```

**Pros:** Simple, explicit, easy to read.
**Cons:** Every write method must remember to call refresh. If someone adds a new write method and forgets, the views go stale silently. The enforcement is by convention, not by structure.

---

**Option B — Spring AOP (Aspect-Oriented Programming)**

You define a cross-cutting aspect that intercepts any method annotated with a custom `@RefreshMaterializedViews` annotation and triggers the refresh automatically after the method completes.

```java
// Custom annotation
@Target(ElementType.METHOD)
@Retention(RetentionPolicy.RUNTIME)
public @interface RefreshMaterializedViews {
    String[] views() default {
        "mv_monthly_expense_summary", 
        "mv_merchant_summary"
    };
}

// Aspect — fires after any annotated method returns successfully
@Aspect
@Component
public class MaterializedViewRefreshAspect {

    private final MaterializedViewRefreshService refreshService;

    @AfterReturning("@annotation(refreshViews)")
    public void refreshAfterWrite(RefreshMaterializedViews refreshViews) {
        refreshService.refreshConcurrently(refreshViews.views());
    }
}

// Usage on the implementation — not the interface
@Override
@Transactional
@RefreshMaterializedViews
public Expense createExpense(UUID userId, CreateExpenseCommand command) {
    // implementation — no refresh call needed here
}
```

**Pros:** Enforcement is structural — if you annotate the method, refresh happens. You cannot forget the refresh call, only the annotation. Adding a new write method is a one-annotation addition. The implementation body stays clean.

**Cons:** AOP is invisible — the refresh happens without any explicit call in the method body. A developer reading `createExpense` for the first time will not immediately see that a refresh happens. This is the standard trade-off with AOP: behaviour is enforced but not visible locally.

---

**Option C — Internal domain events (Observer pattern you already have)**

After every successful write, the service publishes an internal `ExpenseWrittenEvent`. A listener in the infrastructure layer receives the event and triggers the refresh. This is the same Observer pattern defined for the polling-to-push notification upgrade.

```java
@Override
@Transactional
public Expense createExpense(UUID userId, CreateExpenseCommand command) {
    Expense expense = buildExpense(command);
    expenseRepository.save(expense);
    
    // Publish event — no knowledge of what listeners do with it
    eventPublisher.publishEvent(new ExpenseWrittenEvent(expense.getUserId()));
    return expense;
}

// In the infrastructure layer — not in the domain
@Component
public class MaterializedViewRefreshListener {

    @TransactionalEventListener(phase = AFTER_COMMIT)
    public void onExpenseWritten(ExpenseWrittenEvent event) {
        refreshService.refreshConcurrently(
            "mv_monthly_expense_summary",
            "mv_merchant_summary"
        );
    }
}
```

`@TransactionalEventListener(phase = AFTER_COMMIT)` is important — it ensures the refresh only fires after the transaction has committed successfully. If the transaction rolls back, the event is discarded and no stale refresh occurs.

**Pros:** The cleanest separation of concerns. The domain service knows nothing about materialized views. Adding new listeners (notifications, audit events, cache invalidation) requires zero changes to the service. This also wires directly into your future push notification flow — the same event can trigger both a view refresh and an FCM notification.

**Cons:** The most indirection. Tracing what happens after a write requires following the event to its listeners.


**Decision**: Let's try option B with the possibility of evolving to Option C if analysis heavy use case frequency increases
